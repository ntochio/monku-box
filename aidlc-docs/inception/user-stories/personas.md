# User Personas（monku-box PoC）

**前提**: `inception/plans/story-generation-plan.md`（承認済み）、`inception/application-design/application-design.md` の RBAC。

---

## ペルソナ P1 — 匿名投稿者（社内の誰か）

**ロール**: 投稿者（`submitter`）

**目的**: 職場の課題や不満を、身元が特定されにくい形で安全に共有したい。topic ごとに整理された箱に「文句」を入れたい。

**動機**: 直属の上司に言いにくい内容、横串の改善提案、早期の空気読み取りを組織に期待している。

**主要権限（PoC）**: 投稿の作成、投稿フロー内での topic 選択または新規 topic 名の入力。

**典型シナリオ**

- 会議の合間にスマホ／PC から短文を投稿する。
- 手が離せないときは音声で入力し、文字起こしを確認してから送る。
- 音声が使えない・失敗したときはテキストに切り替える。

**フラストレーション**

- 音声が使えないブラウザで「音声だけ」と誘導されること。
- topic を選び忘れて何度もエラーになること。
- 投稿内容がすぐ特定されるような表示になること（PoC ではマスキング・匿名 ID の扱いは実装・要件に従う）。

**成功指標（ペルソナ視点）**

- 数分以内に投稿を完了できる。
- 失敗時も「次に何をすればよいか」が分かる。

## ペルソナ P2 — 閲覧者（管理職・人事・経営など）

**ロール**: 閲覧者（`viewer`）

**目的**: topic 別の傾向や投稿の流れを把握し、対話や施策の判断材料にしたい。重要な変化は通知で気づきたい。

**動機**: 現場の声を定性的に拾い、数値だけでは見えないリスクや機会を早期に知りたい。

**主要権限（PoC）**: ダッシュボード閲覧、**メッセージ一覧**閲覧、通知の受信・一覧から詳細画面への遷移、topic 追加（Application Design に準拠）。

**典型シナリオ**

- 週次でダッシュボードを開き、**円グラフ・棒グラフ**と topic 別の件数や提案文を読む。
- **メッセージ一覧**で個別投稿の本文を topic 絞り込み付きで確認する。
- 通知一覧から該当する分析・一覧画面へ移動する。
- 新しい論点用に topic を追加する。

**フラストレーション**

- ダッシュボードが更新されない／何を見ればよいか分からない。
- 通知が多すぎて優先度が付けられない（PoC では**システムによる優先度付けは行わない** — 人が判断）。

**成功指標（ペルソナ視点）**

- 目的の topic に関する状況を一度の訪問で把握できる。
- 通知から関連画面に迷わず辿れる。

## ペルソナ P3 — 管理者（運用担当）

**ロール**: 管理者（`admin`）

**目的**: アカウントや PoC 範囲の運用設定（辞書・通知ルールの最小更新）を行い、サービスを継続可能な状態に保ちたい。

**動機**: 誤投稿やポリシー違反の抑止、通知の運用品質、利用者追加のオペレーション。

**主要権限（PoC）**: ダッシュボード閲覧、**メッセージ一覧**閲覧、通知受信、ユーザー追加、辞書・通知ルールの**最小**更新（ストーリーマップおよび Application Design に準拠）。

**典型シナリオ**

- 新規の閲覧者／管理者アカウントを登録する。
- 禁止語や通知の閾値など、PoC で用意した最小設定を更新する。
- 閲覧者と同様に通知を受け取り、運用判断をする。
- **メッセージ一覧**で個別投稿を確認し、通報対応やトレンドの裏取りをする。

**フラストレーション**

- 設定変更がコードデプロイ必須で、検証環境と本番の差分が追いにくい（本番移行時の課題としてスコープ外ドキュメントに委ねる）。

**成功指標（ペルソナ視点）**

- PoC 期間中に必要な利用者追加と最小ルール変更が手順に沿って完了する。
- ダッシュボード・**メッセージ一覧**・通知を併用し、運用判断に必要な文脈を得られる。

## ペルソナとストーリーの対応（概要）

| ペルソナ | 主に関わるエピック |
|----------|-------------------|
| P1 匿名投稿者 | エピック 1 |
| P2 閲覧者 | エピック 2（2.1, 2.2, **2.4**）、エピック 3 の topic 追加が閲覧者にも該当する場合 |
| P3 管理者 | エピック 2（2.1, 2.3, **2.4**）、エピック 3 |

詳細なストーリー ID 対応は `stories.md` 末尾の対応表を参照。
