# User Stories（monku-box PoC）

**整理**: Epic-Based（`unit-of-work-story-map.md` と一致）  
**ステータス**: **承認済み**（2026-05-08 — `aidlc-docs/aidlc-state.md` と一致。Story 2.1 の集計脚注・ロードマップ受け入れは **2026-05-10** 追記）  
**受け入れ基準**: 箇条書き（PoC）。チェックは **2026-05-10 時点の PoC 実装**（ユニット `monolith-core`）との照合結果。意図的な範囲外は該当項で明示。

## INVEST の扱い（全ストーリー共通）

| 観点 | PoC での意味 |
|------|----------------|
| **Independent** | 各ストーリーは単体でデモ・テスト可能なユーザー価値を持つ（依存は「前提ロール・データ」に限定）。 |
| **Negotiable** | 画面文言・集計粒度は PoC 範囲で調整可。 |
| **Valuable** | いずれも投稿者・閲覧者・管理者の明示的なニーズに紐づく。 |
| **Estimable** | スコープはストーリーマップと Application Design で境界が定義されている。 |
| **Small** | PoC 単位で 1 スプリント相当に収まる想定。 |
| **Testable** | 各ストーリーに受け入れ基準を記載。 |

## エピック 1 — 投稿者が匿名で投稿する

### Story 1.1 — topic を選択または新規追加して投稿する

**As a** 投稿者（P1）  
**I want** topic を選ぶか新規 topic 名を入力し、**文字起こし（送信前に編集可）欄**に投稿内容を入力して送信したい（直接入力でも可）  
**So that** 職場の論点ごとに文句が整理される

**受け入れ基準**

- [x] 投稿者ロールのみが投稿画面から投稿 API を呼び出せる。
- [x] **既存 topic の選択**または**新規 topic ラベル**のいずれかが必須で、欠けている場合はユーザーに分かるメッセージが表示される。
- [x] 投稿内容が空の場合は投稿できず、エラーまたは案内が表示される。
- [x] 投稿が受理された場合、ユーザーに成功が示される（例: メッセージ表示）。
- [x] topic の存在を事前に厳密検証しない方針のときも、**Policy によるバリデーション**が適用される（ストーリーマップの方向性に一致）。

**INVEST メモ**: 他ストーリーに依存せず、テキストのみでも完結する。

---

### Story 1.2 — 音声入力からテキスト化して投稿する

**As a** 投稿者（P1）  
**I want** 音声を文字に起こしてから投稿したい  
**So that** 入力負担を減らし、すぐに投稿できる

**受け入れ基準**

- [x] 対応ブラウザでは **Web Speech API**（`SpeechRecognition` / `webkitSpeechRecognition`）により日本語（`ja-JP`）の認識セッションを開始でき、**ユーザーが認識を停止**するまで（または API が終了するまで）複数の確定結果を受け取れる。
- [x] **中間結果**を画面上に表示してよい（PoC）。確定結果は編集可能な文字起こし欄に反映される。
- [x] **追記モード**により、投稿前に複数発話を 1 本文に連結できる（要件 Q106）。オフのときは直近の確定結果で文字起こし欄を上書きしてよい。
- [x] 認識結果が画面に反映され、ユーザーが内容を確認したうえで topic とともに送信できる。
- [x] **実際に音声で確定文が取り込まれた投稿**では `inputType` が音声経路として保存データ上区別できる（手打ちのみは `text` — PoC のデータモデルに準拠）。
- [x] **非対応ブラウザ**では、音声開始時にテキスト入力への誘導メッセージが表示され、投稿フローが破綻しない。
- [x] **サーバ経由 STT**（`/v1/api/voice/transcribe`）は PoC では未実装であり、呼び出した場合は **HTTP 501** とエラーコード（例: `NOT_IMPLEMENTED`）が返ることが仕様として許容される — 主経路はブラウザ STT とする。
- [x] 認識中・完了の状態がユーザーに分かる（例: リスニング状態、完了メッセージ）。

**INVEST メモ**: 1.1 の topic／本文ルールに乗るが、音声 UI と STT 経路が独立して検証できる。

---

### Story 1.3 — 失敗時に再試行やテキスト入力に切り替える

**As a** 投稿者（P1）  
**I want** 音声が失敗したときに再試行またはテキスト入力を選びたい  
**So that** 投稿を諦めずに完了できる

**受け入れ基準**

- [x] 音声認識エラー時にユーザーへ失敗が伝わり、**同一の文字起こし欄への直接入力**に切り替えられる（フォールバック）。
- [x] 直接テキスト入力で同じ投稿フロー（topic 必須・Policy）を完走できる。
- [x] 音声が未対応の場合も、テキストのみで Story 1.1 と同等の投稿ができる。
- [x] 再試行で音声セッションを再度開始できる（対応ブラウザの場合）。**認識を停止**後に再度「音声入力を開始」でよい。

**INVEST メモ**: 1.2 のフォールバックとして検証可能。単体では「エラー後もテキストで送れる」でテスト可能。

## エピック 2 — 閲覧者／管理者が状況を把握する

### Story 2.1 — ダッシュボードで topic 別傾向を確認する

**As a** 閲覧者または管理者（P2 / P3）  
**I want** topic 別の傾向や集計をダッシュボードで見たい  
**So that** 現場の声の広がりを把握できる

**受け入れ基準**

- [x] 閲覧者・管理者ロールがダッシュボード API／画面にアクセスできる。投稿者のみのロールでは閲覧できない（または案内のみ）。
- [x] topic 単位で件数または傾向が確認できる（PoC の集計仕様に準拠）。
- [x] **topic 別件数を円グラフ**で確認できる（件数 0 の topic はグラフから除いてよい。凡例で割合を示す）。
- [x] **全メッセージを UTC の暦日（YYYY-MM-DD）で集計した日別件数を棒グラフ**で確認できる（`GET /v1/api/dashboard/summary` の `messagesByDay` 等と整合）。
- [x] 件数の**テキスト一覧**（従来どおり）も併記または併設できる。
- [x] 改善提案など PoC で表示する分析テキストがあれば、ダッシュボードから参照できる。
- [x] **システムによる優先度付け・自動ランキング**は行わない（人が判断 — ストーリーマップの方向性）。
- [x] **件数の意味**が利用者に伝わる: 表示は **受理メッセージの生件数**であり、同一人物の連投・内容の重複は PoC では **別集計しない** 旨を **画面脚注**等で示す（Inception `aggregation-duplicate-and-repeat-submissions.md` §8 承認に整合）。
- [x] **ロードマップ**（ユニーク投稿者系 KPI・重複扱い・業務 ID／Slack）はストーリー本文では実装せず、Application Design §11・提案書 §9 に委ねる（**2026-05-10** 承認）。

**INVEST メモ**: 通知ストーリーと独立して、データがあれば表示検証できる。

---

### Story 2.2 — 通知から対象画面に遷移する

**As a** 閲覧者（P2）  
**I want** 通知一覧から関連する画面へ移動したい  
**So that** 気になった事象をすぐ深掘りできる

**受け入れ基準**

- [x] 閲覧者はアプリ内通知一覧を開ける。
- [x] **PoC スコープ**: 通知行からのディープリンクは実装しない。ホーム等のナビからダッシュボード・メッセージ一覧へ移動し、通知とあわせて状況を把握できることをもって十分とする（将来: 通知ペイロードに応じた直接遷移）。
- [x] **投稿者には通知を送らない**（通知対象は閲覧者・管理者 — ストーリーマップ）。

**INVEST メモ**: 通知レコードとルーティングがあれば単体検証可能。

---

### Story 2.3 — 管理者として通知を受け取り運用判断する

**As a** 管理者（P3）  
**I want** 閲覧者と同様に通知を受け取りたい  
**So that** 運用上の異常や注目イベントに反応できる

**受け入れ基準**

- [x] 管理者は通知一覧にアクセスできる。
- [x] 管理者向け通知の生成条件は Application Design／Functional Design の PoC 範囲に従う。
- [x] 閲覧者と同様、通知一覧・既読およびホームナビによる関連画面への到達で足りる（2.2 の PoC スコープと同じ）。

**INVEST メモ**: ロールが異なるが、2.2 と同じ通知基盤の上で検証できる。

---

### Story 2.4 — メッセージ一覧で投稿を topic 絞り込み付きで確認する

**As a** 閲覧者または管理者（P2 / P3）  
**I want** メッセージ一覧で投稿本文を時系列に見たい（topic で絞り込みたい）  
**So that** 個別の声の内容をダッシュボードの集計とあわせて把握できる

**受け入れ基準**

- [x] 閲覧者・管理者が **メッセージ一覧画面**（例: `/messages`）から `GET /v1/api/messages` を利用できる。投稿者ロールのみのときは閲覧不可または案内のみ。
- [x] **topic フィルタ**（すべて／特定 topic）で一覧を絞り込める（クエリ `topicId` と整合）。
- [x] 各件について表示用本文（例: `bodyDisplay`）、日時、入力種別（テキスト／音声）、topic が識別できる。
- [x] 再読み込み等で最新データを取り直せる。
- [x] **システムによる優先度付け**は行わない（一覧は時系列・フィルタのみ — ストーリーマップの方向性）。

**INVEST メモ**: 2.1（集計）と独立し、メッセージ API と画面があれば検証可能。

## エピック 3 — 管理者が運用設定を行う

### Story 3.1 — アカウント（ユーザー）追加を行う

**As a** 管理者（P3）  
**I want** 閲覧者や他管理者のアカウントを追加したい  
**So that** 適切な人だけがダッシュボード、メッセージ一覧、通知にアクセスできる

**受け入れ基準**

- [x] 管理者のみが PoC で定義された「ユーザー追加」操作を実行できる。
- [x] 投稿者・閲覧者ロールでは当該管理操作ができない。
- [x] 追加後、付与したロールでログイン／ロール切替（PoC の認証方式）が機能する。

**INVEST メモ**: 単一の管理画面／API で完結。

---

### Story 3.2 — 辞書・通知ルールを更新する（PoC 最小）

**As a** 管理者（P3）  
**I want** 禁止語辞書や通知ルールを最小限更新したい  
**So that** PoC でも運用に耐える抑止と通知品質を保てる

**受け入れ基準**

- [x] 管理者のみが PoC で用意された**禁止語辞書**の更新 UI にアクセスできる（通知の生成閾値等の**別 UI によるルール編集**は PoC ではコード／バックエンド定義に委ねる）。
- [x] 辞書更新後、新規投稿への **Policy 適用**に反映される（PoC の永続化方式に準拠）。通知の挙動は実装のイベント定義に従う。
- [x] 閲覧者・投稿者は当該設定変更 UI にアクセスできない。

**INVEST メモ**: 3.1 と独立して、設定ストアとポリシー結合で検証。
