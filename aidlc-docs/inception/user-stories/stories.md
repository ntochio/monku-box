# User Stories（monku-box PoC）

**整理**: Epic-Based（`unit-of-work-story-map.md` と一致）  
**受け入れ基準**: 箇条書き（PoC）  
**関連**: `inception/application-design/unit-of-work-story-map.md`、`application-design.md`

---

## INVEST の扱い（全ストーリー共通）

| 観点 | PoC での意味 |
|------|----------------|
| **Independent** | 各ストーリーは単体でデモ・テスト可能なユーザー価値を持つ（依存は「前提ロール・データ」に限定）。 |
| **Negotiable** | 画面文言・集計粒度は PoC 範囲で調整可。 |
| **Valuable** | いずれも投稿者・閲覧者・管理者の明示的なニーズに紐づく。 |
| **Estimable** | スコープはストーリーマップと Application Design で境界が定義されている。 |
| **Small** | PoC 単位で 1 スプリント相当に収まる想定。 |
| **Testable** | 各ストーリーに受け入れ基準を記載。 |

## エピック 1 — 投稿者が匿名で投稿する

### Story 1.1 — topic を選択または新規追加して投稿する

**As a** 投稿者（P1）  
**I want** topic を選ぶか新規 topic 名を入力して本文を投稿したい  
**So that** 職場の論点ごとに文句が整理される

**受け入れ基準**

- [ ] 投稿者ロールのみが投稿画面から投稿 API を呼び出せる。
- [ ] **既存 topic の選択**または**新規 topic ラベル**のいずれかが必須で、欠けている場合はユーザーに分かるメッセージが表示される。
- [ ] 本文が空の場合は投稿できず、エラーまたは案内が表示される。
- [ ] 投稿が受理された場合、ユーザーに成功が示される（例: メッセージ表示）。
- [ ] topic の存在を事前に厳密検証しない方針のときも、**Policy によるバリデーション**が適用される（ストーリーマップの方向性に一致）。

**INVEST メモ**: 他ストーリーに依存せず、テキストのみでも完結する。

---

### Story 1.2 — 音声入力からテキスト化して投稿する

**As a** 投稿者（P1）  
**I want** 音声を文字に起こしてから投稿したい  
**So that** 入力負担を減らし、すぐに投稿できる

**受け入れ基準**

- [ ] 対応ブラウザでは **Web Speech API**（`SpeechRecognition` または `webkitSpeechRecognition`）により日本語（`ja-JP`）の認識を開始できる。
- [ ] 認識結果が画面に反映され、ユーザーが内容を確認したうえで topic とともに送信できる。
- [ ] 認識成功時に `inputType` が音声経路であることが保存データ上区別できる（PoC のデータモデルに準拠）。
- [ ] **非対応ブラウザ**では、音声開始時にテキスト入力への誘導メッセージが表示され、投稿フローが破綻しない。
- [ ] **サーバ経由 STT**（`/v1/api/voice/transcribe`）は PoC では未実装であり、呼び出した場合は **HTTP 501** とエラーコード（例: `NOT_IMPLEMENTED`）が返ることが仕様として許容される — 主経路はブラウザ STT とする。
- [ ] 認識中・完了の状態がユーザーに分かる（例: リスニング状態、完了メッセージ）。

**INVEST メモ**: 1.1 の topic／本文ルールに乗るが、音声 UI と STT 経路が独立して検証できる。

---

### Story 1.3 — 失敗時に再試行やテキスト入力に切り替える

**As a** 投稿者（P1）  
**I want** 音声が失敗したときに再試行またはテキスト入力を選びたい  
**So that** 投稿を諦めずに完了できる

**受け入れ基準**

- [ ] 音声認識エラー時にユーザーへ失敗が伝わり、**テキスト入力への切り替え**が可能である。
- [ ] テキスト入力で同じ投稿フロー（topic 必須・Policy）を完走できる。
- [ ] 音声が未対応の場合も、テキストのみで Story 1.1 と同等の投稿ができる。
- [ ] 再試行で音声を再度開始できる（対応ブラウザの場合）。

**INVEST メモ**: 1.2 のフォールバックとして検証可能。単体では「エラー後もテキストで送れる」でテスト可能。

## エピック 2 — 閲覧者／管理者が状況を把握する

### Story 2.1 — ダッシュボードで topic 別傾向を確認する

**As a** 閲覧者または管理者（P2 / P3）  
**I want** topic 別の傾向や集計をダッシュボードで見たい  
**So that** 現場の声の広がりを把握できる

**受け入れ基準**

- [ ] 閲覧者・管理者ロールがダッシュボード API／画面にアクセスできる。投稿者のみのロールでは閲覧できない（または案内のみ）。
- [ ] topic 単位で件数または傾向が確認できる（PoC の集計仕様に準拠）。
- [ ] 改善提案など PoC で表示する分析テキストがあれば、ダッシュボードから参照できる。
- [ ] **システムによる優先度付け・自動ランキング**は行わない（人が判断 — ストーリーマップの方向性）。

**INVEST メモ**: 通知ストーリーと独立して、データがあれば表示検証できる。

---

### Story 2.2 — 通知から対象画面に遷移する

**As a** 閲覧者（P2）  
**I want** 通知一覧から関連する画面へ移動したい  
**So that** 気になった事象をすぐ深掘りできる

**受け入れ基準**

- [ ] 閲覧者はアプリ内通知一覧を開ける。
- [ ] 各通知から、関連する画面（例: ダッシュボード／該当 topic）へ遷移できる、または同等の情報に到達できる。
- [ ] **投稿者には通知を送らない**（通知対象は閲覧者・管理者 — ストーリーマップ）。

**INVEST メモ**: 通知レコードとルーティングがあれば単体検証可能。

---

### Story 2.3 — 管理者として通知を受け取り運用判断する

**As a** 管理者（P3）  
**I want** 閲覧者と同様に通知を受け取りたい  
**So that** 運用上の異常や注目イベントに反応できる

**受け入れ基準**

- [ ] 管理者は通知一覧にアクセスできる。
- [ ] 管理者向け通知の生成条件は Application Design／Functional Design の PoC 範囲に従う。
- [ ] 通知から関連画面へ辿れる（2.2 と同様のナビゲーション品質）。

**INVEST メモ**: ロールが異なるが、2.2 と同じ通知基盤の上で検証できる。

## エピック 3 — 管理者が運用設定を行う

### Story 3.1 — アカウント（ユーザー）追加を行う

**As a** 管理者（P3）  
**I want** 閲覧者や他管理者のアカウントを追加したい  
**So that** 適切な人だけがダッシュボードや通知にアクセスできる

**受け入れ基準**

- [ ] 管理者のみが PoC で定義された「ユーザー追加」操作を実行できる。
- [ ] 投稿者・閲覧者ロールでは当該管理操作ができない。
- [ ] 追加後、付与したロールでログイン／ロール切替（PoC の認証方式）が機能する。

**INVEST メモ**: 単一の管理画面／API で完結。

---

### Story 3.2 — 辞書・通知ルールを更新する（PoC 最小）

**As a** 管理者（P3）  
**I want** 禁止語辞書や通知ルールを最小限更新したい  
**So that** PoC でも運用に耐える抑止と通知品質を保てる

**受け入れ基準**

- [ ] 管理者のみが PoC で用意された辞書・通知ルールの更新手段にアクセスできる。
- [ ] 更新後、新規投稿への Policy 適用または通知生成に反映される（PoC の永続化方式に準拠）。
- [ ] 閲覧者・投稿者は当該設定変更 UI にアクセスできない。

**INVEST メモ**: 3.1 と独立して、設定ストアとポリシー結合で検証。
