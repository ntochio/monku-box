# 要件メモ：同一人物の連投・重複投稿と集計結果の歪み

**種別**: Inception／要件分析（追加論点）  
**状態**: **承認済み**（2026-05-10 — ユーザー回答・計画承認。`aidlc-state.md` を参照）  
**関連**: `application-design.md`（Analysis・ダッシュボード）、`construction/monolith-core/functional-design/`、`src/application/dashboard-summary.ts`、`src/application/submit-message.ts`

---

## 1. 懸念の要約

- **同一人物**が短時間に複数回投稿すると、**件数ベースの集計**（topic 別件数、日別件数、提案の根拠となる `count` 等）が **「声の多さ」ではなく「操作回数」** に引きずられる可能性がある。
- **重複投稿**（誤操作の再送、同一内容のコピペ、ボット的な連投）が、ダッシュボードや改善提案の **信頼性・公平性** に影響しうる。
- 一方で、**匿名性**・**監視されない安心感** を売りにするプロダクトでは、**個人を強く紐づける**対策は受け入れられない場合がある。

本資料は **トレードオフを明示**し、PoC と本番で **何を採用するか** を決めるためのたたき台である。

---

## 2. 現行実装の事実（2026-05-10 時点）

| 項目 | 内容 |
|------|------|
| 集計の定義 | `buildDashboardSummary` は **メッセージ 1 件 = カウント 1**（topic 別・UTC 日別とも）。重複・同一投稿者の扱いは **なし**。 |
| 匿名投稿者 ID | `MessageRecord.anonymousSubmitterId` あり。API は `anonymousSubmitterId` を **任意**で受け取るが、**投稿フォームは未送信**のため、未指定時は **毎回新規 ID** が採番される。 |
| 同一人物の追跡 | **ブラウザを跨いだ同一人物の識別は PoC UI では実質できない**（クライアントが安定 ID を送らない限り）。 |
| 重複検知 | **同一本文・近時刻**の検知やマージは **未実装**。 |

→ 集計は **「受理された Message 件数」** をそのまま可視化している。連投・重複は **件数にそのまま乗る**。

---

## 3. 整理用語

| 用語 | 意味の例 |
|------|----------|
| **連投** | 同一人物（または同一セッション）による短時間の複数投稿。悪意なくても件数が増える。 |
| **重複投稿** | 内容が同一または極めて近い投稿が複数レコードとして残る状態。 |
| **集計の単位** | 「生メッセージ件数」「ユニーク投稿者加重」「重複除外後件数」など、ダッシュボードが何を見せるかの定義。 |

---

## 4. 対策の方向性（選択肢）

実装・プライバシー・UX のバランスで組み合わせる。

| ID | 方針 | メリット | デメリット／注意 |
|----|------|----------|------------------|
| A | **現状維持**（件数 = メッセージ数） | 実装・説明が単純。 | 連投・重複で歪む。 |
| B | **クライアント安定 ID**（first-party cookie / localStorage 等で `anonymousSubmitterId` を再利用） | 同一ブラウザ内の連投を **識別可能**。集計を「ユニーク投稿者数」や「投稿者あたり上限」に拡張しやすい。 | 端末共有・Cookie 削除で分断。プライバシー説明が必要。 |
| C | **レート制限**（IP・セッション・投稿者 ID 単位の時間あたり上限） | 乱発を抑止。 | 匿名 PoC では IP の取り扱い・共有 NAT の誤爆に注意。 |
| D | **重複検知**（本文ハッシュ、類似度、時間窓内の同一 topic 等） | 誤二重投稿を集計から除きやすい。 | 誤検知（正当な似た声）とコスト。PoC ではルールが難しい。 |
| E | **表示の二層化** | 「総投稿数」と「推定ユニーク声」等を **併記**し、歪みを運用で解釈。 | UI・説明コスト。 |

## 5. 提案（ドキュメント上の初期スタンス）

- **PoC**: 集計定義を **「受理メッセージ件数」** と明記し、連投・重複による歪みは **既知の制約** として README／ダッシュボード脚注で説明する、が **最小コスト**。  
- **本番前**: B（安定 ID）と C（レート制限）の **どちらを主とするか**、および D をどこまでやるかを、プライバシー方針とセットで決める。

※ 確定案ではない。**§6 の回答**で上書きする。
