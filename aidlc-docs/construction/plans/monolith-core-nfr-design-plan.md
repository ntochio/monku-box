## 確認質問（NFR Requirements Plan の回答を [Answer] として取り込み）

設計判断の根拠は `monolith-core-nfr-requirements-plan.md` の Q1〜Q8 に準拠する。以下は NFR Design 観点での補足確認（PoC では明示回答を設計に反映済みとする）。

### Q-D1. レジリエンス（障害・再試行）

障害時の基本方針と、クライアント／サーバの再試行の扱いは？

[Answer]: サーバ側は同一リクエスト処理を **単一トランザクション** とし、失敗時は **ロールバック** して不整合を残さない。PoC では **サーキットブレーカー／バックグラウンドリトライ基盤は導入しない**。利用者は **手動再試行**（Functional Design の方針と一致）。

### Q-D2. スケーラビリティ・負荷分離

水平スケール、キュー、非同期ワーカー等の採用は？

[Answer]: **採用しない**。単一デプロイ／固定キャパシティ前提（Q1）。将来、負荷分離が必要になった場合にキュー等を再検討。

### Q-D3. 性能最適化パターン

キャッシュ層、CDN、プリコンピュートの要否は？

[Answer]: **明示キャッシュ層・CDN は PoC では必須としない**。ダッシュボード集計は同期処理で十分な範囲に留める。ボトルネックが出た場合のみ局所最適化（Q2）。

### Q-D4. セキュリティ実装パターン

認可・秘密情報・入力の実装レベルは？

[Answer]: **RBAC は Application Design に従いコードパスで強制**。秘密情報は **環境変数等** に限定しリポジトリに含めない。**入力バリデーションは Policy 集約**（既存）。脅威モデリング・WAF・正式コンプライアンスは対象外（Q4）。

### Q-D5. 論理コンポーネント（キュー・キャッシュ等）

PoC で論理アーキテクチャに含める補助コンポーネントは？

[Answer]: **メッセージキュー・分散キャッシュ・全文検索基盤は持たない**。永続は **単一ストア**（ファイルまたは軽量 DB のいずれか）。通知は **アプリ内ストア＋即時生成**（Q7）。

`requirement-verification-questions.md` の **「インフラ構築および通信方式について（PoC）」** に従い、論理コンポーネント図は **JSON 中心の HTTP 連携**を前提とし、SOAP/gRPC 等の確定は行わない。
