## 確認質問（既存回答反映）

### Q1. スケーラビリティ要件
[Answer]: PoC ではオートスケールを必須にしない。固定キャパシティで運用し、本番移行時に再検討。

### Q2. パフォーマンス要件
[Answer]: 数値SLOは設定しない。「ものすごく遅くなければOK」を受け入れ基準にする。

### Q3. 可用性要件
[Answer]: PoC はベストエフォート。停止時は手動復旧を許容。

### Q4. セキュリティ要件
[Answer]: Security Baseline は無効。PoC では必須化しないが、余力があれば最低限の入力バリデーション・秘密情報管理を行う。

### Q5. 信頼性/整合性要件
[Answer]: 処理失敗時はロールバックして不整合回避。投稿/分析更新/通知生成は同一処理単位を優先。

### Q6. 運用監視/ログ要件
[Answer]: 本格運用監視・ログモニタリングはスコープ外。最小ログのみ。

### Q7. 通知NFR
[Answer]: アプリ内通知のみ、即時配信、ノイズ制御はPoCで実装しない。受信対象は閲覧者と管理者、投稿者は対象外。

### Q8. データ永続性要件
[Answer]: PoC は軽量運用（CSV/JSON許容）。高度なバックアップ/インデックス最適化は必須化しない。

`requirement-verification-questions.md` の **「インフラ構築および通信方式について（PoC）」** に従い、NFR 成果物では **インフラの本格確定**および **SOAP/gRPC 等の通信標準の厳密化**を PoC で求めない一方、**JSON 中心の API** と **差し替え可能な境界**を保守性・相互運用の観点で記載する。
