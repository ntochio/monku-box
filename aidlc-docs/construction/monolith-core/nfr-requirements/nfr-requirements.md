# NFR Requirements - monolith-core

## 1. NFR 方針（PoC）

本ユニットの NFR は、**PoC 完走とデモ再現性**を優先し、以下の境界で定義する。

- 本番品質の非機能（監査、高可用、本格監視、厳密性能最適化）は将来検討
- ただし、機能破綻を避けるための最小信頼性（整合性/基本エラー処理）は実施

## 2. 観点別要件

### 2.1 Scalability（拡張・スケール）
- オートスケール要件: **対象外（PoC）**
- スケーリング戦略: 単一構成/固定キャパシティで運用
- 再検討タイミング: 本番移行判断時

### 2.2 Performance（性能）
- 数値目標（p95/p99等）: **未設定**
- 受け入れ基準: 「ものすごく遅くなければよい」
- ボトルネック調整: 問題発生時の局所対応

### 2.3 Availability（可用性）
- 目標稼働率: 未設定（ベストエフォート）
- 障害時対応: 手動復旧許容
- デプロイ時停止: 短時間停止を許容

### 2.4 Security（セキュリティ）
- Security Baseline: 無効（PoC方針）
- 必須実施:
  - 最低限の入力バリデーション
  - 秘密情報（キー等）の平文露出回避
- 高度要件（脅威モデル/監査証跡/正式対応プロセス）: 将来検討

### 2.5 Reliability（信頼性・整合性）
- 失敗時はロールバックし、整合性崩壊を回避
- 投稿、分析更新、通知イベント生成は同一処理単位を優先
- クライアント手動再試行を基本とする

### 2.6 Observability（可観測性）
- 本格運用監視/ログモニタリング: **対象外（PoC）**
- 実装: 最小限のログ出力のみ

### 2.7 Maintainability（保守性）
- Frontend は DDD 的分離（presentation/application/domain/infrastructure）を採用
- API は `/v1/api/...` としてバージョン管理
- 優先付けロジックは実装しない（人判断）

### 2.9 Interoperability / 通信（PoC）

- **ペイロード形式**: クライアントとサーバの主要な API は **JSON** を中心とする（`requirement-verification-questions.md` 冒頭「インフラ構築および通信方式について」に準拠）。
- **プロトコル標準**: SOAP、gRPC、GraphQL 等の**全面採用方針の確定**は PoC では行わない。HTTP 上の JSON を当面の実装基盤とし、本番移行時に再評価する。
- **拡張性**: インフラの本格確定は行わないが、**外部連携・永続化は差し替え可能な境界**（アダプタ等）を推奨し、将来のプロトコル/ホスト変更に備える。

### 2.8 Usability（使いやすさ）
- 投稿フロー:
  - topic 必須（既存選択または新規追加）
  - 音声入力は送信前確認/編集を必須化
- 通知:
  - 受信対象は閲覧者・管理者
  - 投稿者は通知対象外

## 3. NFR 受け入れ基準（PoC）

1. 主要ユースケース（投稿/閲覧/通知/管理）が致命停止なく動作する
2. 失敗時に整合性不一致を残さない
3. 操作不能な性能劣化が発生しない
4. 本番向け NFR が未対応であることを明示できる

## 4. 明示的非対象（PoC）

- SLA/SLO の正式定義
- 監査証跡の厳密要件
- 24/7 運用前提の監視/アラート
- オートスケール/多AZなどの本格可用性設計
- パフォーマンスチューニングの網羅
- インフラ構築の本格確定、通信方式（SOAP/gRPC 等）の厳密な標準化の完了（実装は JSON 中心・境界分離で拡張余地を確保）
