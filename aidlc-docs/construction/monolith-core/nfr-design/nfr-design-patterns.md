# NFR Design Patterns - monolith-core (PoC)

## 1. 目的

`nfr-requirements.md` で定義した PoC 向け NFR を、**実装可能な設計パターン**に落とし込む。本番向けパターン（高可用、厳密 SLO、本格監視）は **明示的に採用しない**。

## 2. パターン一覧

| パターン | PoC での適用 | 概要 |
|----------|----------------|------|
| **単一ユニット・オブ・ワーク（同期）** | 採用 | 投稿受理〜（方針どおり）分析更新・通知生成までを **同一処理単位** で完結させ、途中失敗時は **全体をロールバック** |
| **手動再試行（クライアント主導）** | 採用 | 自動リトライ基盤は置かず、UI からの再実行を基本 |
| **フェイル・クローズ（PoC 最小）** | 採用 | 外部 STT 等が失敗した場合は **エラーを返し**、音声以外の経路（テキスト）へ誘導 |
| **API バージョニング** | 採用 | `/v1/api/...` プレフィックスで破壊的変更時の移行余地を確保 |
| **フロント DDD レイヤ分離** | 採用 | presentation / application / domain / infrastructure により変更局所化 |
| **最小観測性（ログ）** | 採用 | リクエスト単位の識別子・エラー理由など **最小限** のログ。ダッシュボード型監視は対象外 |
| **秘密情報の外部化** | 採用 | API キー等は環境変数またはデプロイ環境のシークレットに限定 |
| **サーキットブレーカー** | 不採用 | PoC では導入コスト対効果が低い |
| **メッセージキュー＋ワーカー** | 不採用 | 非同期分離は本番移行時に再評価 |
| **分散キャッシュ / CDN** | 不採用（必須化しない） | 遅延が問題になった場合のみ局所検討 |
| **オートスケール** | 不採用 | NFR どおり固定キャパシティ |

## 3. 観点別の設計方針

### 3.1 信頼性・整合性

- **トランザクション境界**: 永続化レイヤがサポートする範囲で、投稿とそれに付随する一貫性が必要な更新を **まとめてコミット** する。
- **部分成功の禁止**: 分析・通知を投稿と同一単位に含める方針のため、「投稿だけ成功し通知だけ失敗」のような状態を **許容しない**（失敗時はロールバック）。

### 3.2 性能

- 数値目標は持たない。実装上の注意:
  - ダッシュボードは **必要最小のクエリ** に留める（全件スキャンを常態化させない程度の配慮）。
  - N+1 が明らかな箇所は **まとめ取得** を検討（過度な最適化はしない）。

### 3.3 セキュリティ（PoC 最小）

- **RBAC**: ルートハンドラ／サーバアクションの入口でロールを検証。
- **入力**: Content Policy Engine に集約されたバリデーション・マスキングを通過させる。
- **Security Baseline**: 無効のまま。本番移行時に拡張。

### 3.4 可用性

- 目標稼働率なし。デプロイ時の短時間停止を許容。
- マルチ AZ・ヘルスチェック連動の自動復旧は設計しない。

### 3.5 ユーザビリティ（NFR との接続）

- 音声入力は **送信前の確認・編集** を必須とし、STT 失敗時はテキストへフォールバック可能にする（`nfr-requirements.md` 2.8 と整合）。

## 4. 明示的非採用（再掲）

`nfr-requirements.md` 4 節と同趣旨。本設計段階でも **SLO 管理、監査証跡の厳密化、本格 APM、オートスケール** は扱わない。
