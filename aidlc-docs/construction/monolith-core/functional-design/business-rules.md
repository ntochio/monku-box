# Business Rules - monolith-core

## 1. 投稿ルール

1. 1 投稿には 1 topic を必須とする。
2. 投稿時に topic は「既存選択」または「新規追加」を許容する。
3. 新規 topic が既存と完全一致する場合は既存 topic を採用する（重複抑止、Q1=B）。
4. 投稿者は投稿作成・topic 選択/追加を実行可能。

## 2. Policy バリデーションルール

1. バリデーション責務は Content Policy Engine に集約する。
2. 必須チェック、文字数上限/下限チェックを実施する（Q80=A,B）。
3. NG 判定時は投稿拒否ではなく、マスク/補正して受理する（Q2=B）。
4. topic 存在検証は行わない（運用上、追加/紐付けフローで吸収）。

## 3. 音声入力ルール

1. STT は **Web Speech API**（ブラウザ）を主経路とする（Q4=A）。`POST /v1/api/voice/transcribe` は PoC では使用しない（差し替え用・501）。
2. **音声入力を開始** / **認識を停止**でセッションを制御する。`continuous` + `interimResults` により複数の確定結果を扱い、投稿前に**追記**または**上書き**で蓄積できる（Q106=B）。
3. 送信前に文字起こし結果を必ず編集可能にする（Q105=A）。
4. topic は送信時点で必須（既存選択または新規）。音声と topic の UI 順序は PoC で柔軟（Q107=B,C と整合するよう、候補提示は将来拡張）。
5. マイク拒否・`no-speech`・`not-allowed` 等はユーザー向けメッセージで案内し、テキスト入力へフォールバックする（Q109）。

## 4. 通知ルール

1. 通知対象イベントは「新規投稿」「topic 急増」（Q3=A,B）。
2. 通知チャネルはアプリ内のみ（Q114=A）。
3. 通知は即時配信（Q115=A）。
4. PoC ではノイズ抑制ロジックを入れない（Q116=E）。
5. 通知本文は件名レベルのみ（Q117=A）。
6. 通知受信対象は閲覧者・管理者。投稿者は対象外。

## 5. ダッシュボード表示ルール

1. 必須表示は topic 別件数と改善提案カード（Q6=A,D）。
2. 改善提案は topic 単位で生成する（Q98=A）。
3. 提案形式は短文提案 + 根拠データとする（Q99=A,B）。
4. 提案は AI 提案としてそのまま表示する（Q100=A）。

## 6. 権限ルール（RBAC）

1. 投稿者
   - 投稿作成
   - 自分の投稿履歴閲覧
   - topic 選択/提案
2. 閲覧者
   - ダッシュボード閲覧
   - 匿名化済みメッセージ閲覧
   - topic 管理（追加・統合・アーカイブ）
   - 通知受信
3. 管理者
   - 閲覧者権限を含む
   - アカウント追加
   - 辞書更新（PoC 最小）
   - 通知受信

## 7. データ整合ルール

1. 処理失敗時はロールバックし、整合性が崩れる状態を残さない（Q81 補足）。
2. 投稿・分析・通知イベントは同一処理単位で完了させる（Q7=A）。
3. 非同期化は将来最適化とする。

## 8. 非目標ルール（PoC）

1. 優先付け・優先度スコアリングはシステム実装しない。
2. 定期レポート配布は実施しない（ダッシュボード確認を主軸）。
3. 監査、法務、運用監視、本格ログモニタリングは将来検討。**拡張性の精緻な設計ドキュメントの完成**も PoC では必須としない（ただし実装は要件確認書のとおり **将来の拡張に備えた境界**を推奨）。
4. DB スキーマ厳密管理は PoC では必須化しない（CSV/JSON 含む軽量運用を許容）。

## 9. 通信・インフラ（要件確認書との整合）

1. **インフラの本格確定**（クラウド/IaC/ネットワークの完成度）および **通信標準の厳密確定**（SOAP、gRPC 等の全面方針）は PoC では行わない。
2. **API のデータ表現は JSON を中心**とする（`requirement-verification-questions.md`「インフラ構築および通信方式について」）。
3. 外部 STT/LLM/永続化は **アダプタ**で抽象化し、プロトコル・ホスト変更が局所に閉じるよう実装する。
