# Business Logic Model - monolith-core

## 1. 対象ユースケース

- UC-A: 投稿（テキスト）
- UC-B: 投稿（音声入力）
- UC-C: ダッシュボード閲覧
- UC-D: 管理者アカウント追加

本書は PoC 向けの機能ロジックを定義する。監査・運用監視・本格性能最適化は対象外。API 連携の主要なデータ表現は **JSON** とし、インフラ・通信標準（SOAP/gRPC 等）の厳密確定は PoC では行わない（`requirement-verification-questions.md`「インフラ構築および通信方式について」）。

## 2. UC-A 投稿（テキスト）

### 入力
- 投稿本文
- topic 情報（既存 topic 選択 または 新規 topic 追加）
- 投稿者コンテキスト（ロール、匿名内部 ID 付与対象）

### 処理フロー
1. API が投稿者権限を確認する。
2. topic 情報を正規化する。
   - 既存 topic 指定時: その topic を採用
   - 新規 topic 文字列指定時: 完全一致 topic を検索
   - 完全一致がある場合は既存 topic を採用（Q1=[Answer]: B）
   - 完全一致がない場合は topic を新規作成
3. Content Policy Engine がバリデーションを実施する。
   - 必須チェック
   - 文字数上限/下限チェック（Q80=[Answer]: A,B）
   - マスキング/補正ルール適用
4. Policy 判定が NG の場合でも、補正後データとして投稿を受理する（Q2=[Answer]: B）。
5. Message を保存する。
6. 同一トランザクション内で分析更新・通知イベント生成を実行する（Q7=[Answer]: A）。

### 出力
- 投稿受理結果
- 正規化済み topic 参照
- 表示用本文（必要に応じ補正済み）

## 3. UC-B 投稿（音声入力）

### 入力
- 音声データ
- 投稿者コンテキスト

### 処理フロー
1. フロントで **Web Speech API** の認識セッションを**開始**し、ユーザーが**認識を停止**するまで発話を受け付ける（Q106=[Answer]: B）。中間結果の表示を許容する。
2. ブラウザ内で STT を実行（Q4=[Answer]: A）。サーバ `/v1/api/voice/transcribe` は PoC では使用しない。
3. 確定した文字起こしを送信前に必ずユーザーに提示し、編集可能状態にする（Q105=[Answer]: A）。**追記**により複数発話を連結できる。
4. topic は送信時点で必須（既存または新規）。音声内容からの topic 候補提示は将来拡張（Q107=[Answer]: B,C）。
5. 以降は UC-A と同じ投稿処理へ合流する。

### フォールバック
- STT 失敗時はテキスト入力へ切替（既存設計準拠）。
- AI 整形失敗時はフォールバック文言で受理（Q79=[Answer]: D）。

## 4. UC-C ダッシュボード閲覧

### 入力
- 閲覧者/管理者ロール
- フィルタ条件（topic、期間）

### 処理フロー
1. API が閲覧ロール（閲覧者・管理者）を確認。
2. 投稿データから topic 別件数を集計する（Q6=[Answer]: A）。
3. 改善提案カードを生成/取得して表示する（Q6=[Answer]: D）。
   - 提案単位は topic ごと（Q98=[Answer]: A）
   - 提案形式は短文 + 根拠データ（Q99=[Answer]: A,B）
   - 提案は AI 提案としてそのまま表示（Q100=[Answer]: A）
4. 通知一覧リンクを通じて新規投稿・急増通知に遷移する。

## 5. UC-D 通知生成・受信

### 通知生成条件
- 新規投稿（Q3=[Answer]: A）
- topic 急増（Q3=[Answer]: B）

### 通知配信
- チャネル: アプリ内通知のみ（Q114=[Answer]: A）
- 頻度: 即時（Q115=[Answer]: A）
- ノイズ制御: PoC では制御なし（Q116=[Answer]: E）
- 表示粒度: 件名レベル（Q117=[Answer]: A）

### 通知受信ロール
- 受信対象: 閲覧者・管理者
- 投稿者は対象外

## 6. UC-E 管理者アカウント追加

### 入力
- 追加対象のメール
- 付与ロール

### ルール
- 管理者のみ実行可能（RBAC）
- ロール指定は必須（Q5=[Answer]: B）
- PoC では最小実装（監査証跡は将来検討）

## 7. トランザクション方針

- 整合性優先で失敗時はロールバックし、不整合状態を残さない（Q81=[Answer]: D 補足コメント）。
- 投稿保存・分析更新・通知イベント生成は同一処理内で実施（Q7=[Answer]: A）。

## 8. 完了条件トレース

- 主要ユースケース 3 本以上のルールを明文化（Q8=[Answer]: B）
- RBAC と Policy 責務の矛盾なし（Q8=[Answer]: C）
