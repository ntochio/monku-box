# Infrastructure Design - monolith-core (PoC)

## 0. 要件正本との関係

`requirement-verification-questions.md` の **「インフラ構築および通信方式について（PoC）」** に従い、本書に記すクラウド/PaaS/単一ストア等は **PoC を動かすための例示・最小マッピング**であり、**インフラの本格確定や通信標準（SOAP/gRPC 等）の完成**を意味しない。本番移行時にインフラ・通信を再オープンする。

## 1. 目的

論理コンポーネント（`nfr-design/logical-components.md`）を、**PoC で現実的にプロビジョニング可能なインフラ選択**にマッピングする。本番レベルの冗長化・本格監視・オートスケールは **スコープ外**（`nfr-requirements.md`）。

## 2. 論理 → インフラ対応表

| 論理コンポーネント | PoC でのインフラ実体 | 備考 |
|--------------------|----------------------|------|
| Web クライアント | 利用者ブラウザ | STT は主にクライアント機能 |
| Next.js モノリス（UI + API + ドメイン） | **単一 Node.js ランタイム**上の Next.js アプリ | ローカルまたは PaaS の 1 サービス |
| HTTP API 境界 | Next.js Route Handlers 等（同一プロセス） | 別途 API Gateway は置かない |
| 永続ストア | **SQLite ファイル** または **アプリ配下の JSON/CSV**（チーム選定） | 単一ボリューム／単一ファイル想定 |
| Content Policy / 分析 / 通知 | アプリプロセス内モジュール | 外部キューなし |

## 3. クラウド・プロバイダ

- **ベンダーロックを PoC で強制しない**。ホスティングは「Next.js が動く単一環境」であれば可。
- 参考構成: ローカル開発 +（任意）**単一リージョンのマネージド Node ホスティング**。

## 4. コンピュート

| 項目 | PoC |
|------|-----|
| スケール | 固定（1 インスタンス想定） |
| オートスケール | なし |
| コンテナ | 必須ではない（チームが Docker を使っても可） |

## 5. ストレージ・データライフサイクル

| 項目 | PoC |
|------|-----|
| バックアップ | 自動スナップショット等は **必須化しない** |
| 保持期間 | デモ完走に足る範囲（運用ポリシーは未定義でよい） |
| 暗号化（ディスク） | PaaS 既定があれば従う。追加要件は定めない |

## 6. メッセージング・非同期

- **未使用**。将来、投稿と通知を分離する場合に再検討。

## 7. ネットワーク

| 項目 | PoC |
|------|-----|
| 入口 | HTTPS（ホスト/PaaS 設定に従う） |
| ロードバランサ | PaaS 既定があれば利用。独自設計はしない |
| WAF / DDoS 緩和 | 必須化しない（Security Baseline 無効と整合） |

## 8. 秘密情報・設定

| 種類 | 扱い |
|------|------|
| セッション／将来の API キー | **環境変数** またはホストのシークレット機能 |
| データファイルパス | 環境変数で上書き可能にしてもよい（任意） |

## 9. 監視・ログ

- **APM・集中ログ基盤は導入しない**。
- アプリ: リクエスト失敗・例外の **最小ログ**。
- ホスト: 利用可能なら **標準ログ閲覧** に限定。

## 10. 共有インフラ

- **なし**（本ユニット単独）。マルチテナント分離のインフラ要件は定義しない。

## 11. 将来（本番移行時）の見直し項目

- RDB マネージドサービスへの移行、バックアップ・リストア手順
- Security Baseline 準拠、WAF、シークレットローテーション
- 可用性目標に応じた冗長化・ヘルスチェック
