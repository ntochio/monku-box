# Deployment Architecture - monolith-core (PoC)

## 1. 目的

**単一アプリケーション**としての配置と、開発〜デモまでの **環境の考え方** を定義する。詳細な IaC（Terraform 等）は PoC では **必須としない**。

`requirement-verification-questions.md`（**インフラ構築および通信方式について**）に従い、デプロイ手順・ホスト選定は **深く確定しない**。将来のホスト変更に耐えうるよう、手順は最小限の記述に留める。

## 2. 環境区分

| 環境 | 用途 | PoC での想定 |
|------|------|----------------|
| **Local** | 実装・手動検証 | 開発者マシン上の Next.js |
| **Demo / Shared**（任意） | ステークホルダー向けデモ | 1 つの共有 URL（PaaS 等） |

ステージング／本番の厳密分離は **定義しない**。

## 3. 配置の流れ（テキスト）

- 利用者ブラウザが **HTTPS** で **Next.js モノリス（単一インスタンス）** に接続する。
- アプリが **単一ローカルストア**（SQLite または JSON/CSV 等のファイル）を読み書きする。
- **ブラウザ STT** は主にクライアント内で完結し、テキストとして UI に渡す（追加のサーバホップなし）。
- **サーバ経由 STT** を将来使う場合も、**同一 Next.js** の `/v1/api/...` ルートに収める。

利用者から見える **オリジンは 1 つ**。API は同一プロセスの Route Handlers。データストアは **デプロイに近接した単一ストア**。

## 4. デプロイフロー（論理）

1. ソースを取得し依存関係をインストール
2. 環境変数を設定（シークレット・データパス等）
3. `next build` 相当のビルド
4. 単一プロセスで起動（`next start` またはホスト既定）

**ロールバック**: PoC では手順を厳密化しない。問題時は **前バージョンを再デプロイ** を基本とする。

## 5. ネットワーク・エンドポイント

| エンドポイント | 公開 |
|----------------|------|
| Web UI + `/v1/api/*` | インターネット向け（デモ時）または localhost のみ |
| 管理用 DB ポート | **公開しない**（ファイル/SQLite はアプリと同ホスト） |

## 6. 依存サービス（外部）

| 依存 | 必須 | 備考 |
|------|------|------|
| ブラウザ STT | PoC 優先経路 | クライアント側 |
| サーバ STT / LLM | 任意 | 利用時のみ API キーを環境変数で渡す |

外部呼び出しが失敗した場合の挙動は **アプリのエラーハンドリング** に従う（NFR のフェイル・クローズ）。

## 7. 非目標（PoC）

- ブルーグリーン／カナリアの手順書
- マルチ AZ
- 専用 CI/CD パイプラインの必須化

本番移行時に `infrastructure-design.md` の「将来見直し」と合わせて再設計する。
